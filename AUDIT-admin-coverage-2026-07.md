# Homy — Покрытие API админкой (что нельзя сделать без БД/костылей)

**Вопрос:** всё, что умеет API (и платформа), должно делаться через админку — ничего через БД или прямые запросы.
**Ответ коротко:** админка есть, но покрывает **~6 действий из десятков**. Большинство управляющих операций сейчас возможны только через БД или прямой вызов API — это костыль. Ниже — полная честная карта.

---

## 0. Что админка умеет СЕЙЧАС (весь её объём)

Есть **две** админ‑поверхности, обе дергают одни и те же 4 роута (это ещё и дубль, требует консолидации):
- новая `components/homy/AdminPanel.tsx` (рендерится в `/dashboard` для админа) — Обзор / Модерация / Пользователи;
- старая `app/admin/*` (`page.tsx`, `moderation/page.tsx`, `users/page.tsx`, `layout.tsx`).

Всего админских роутов — **5**: `admin/dashboard` (GET), `admin/listings` (GET), `admin/listings/[id]/approve` (POST), `admin/listings/[id]/reject` (POST), `admin/users` (GET, PATCH).

Реальные возможности админа:
1. Смотреть метрики платформы.
2. Смотреть очередь модерации (`status=pending`).
3. **Approve / Reject** поданного объявления.
4. Смотреть список пользователей (фильтр role/type/blocked).
5. **Блок / разблок** пользователя.
6. Сменить **`role`** (user / moderator / admin).

**Всё.** Ниже — что API/платформа умеет ещё, но через админку сделать нельзя.

## Вывод

Админка покрывает ~6 действий; платформе нужно ~25+ управляющих операций. Всё, что не в списке §0, сейчас делается **через БД или прямой API** — это и есть костыли, которые ты хочешь убрать.

Причём часть возможностей **уже есть в API** — их нужно просто дать админу (UI + разрешить админский вызов): редактирование/снятие/удаление объектов, редактирование заявок. А часть **API‑роутов вообще нет** — их нужно построить: смена `user_type`/провижининг консультанта, модерация отзывов, админский обзор просмотров/лидов/сделок, просмотр waitlist, авторинг туров, broadcast‑уведомления.

## Предлагаемый порядок (закрываем костыли)

1. 🔴 **Управление пользователями** (закрывает дыру после C1): в `admin/users` PATCH добавить `set_user_type` и «назначить/снять консультанта»; в UI «Пользователи» — селектор персоны + действие «Сделать консультантом». *(Без этого консультанта нельзя создать иначе как БД.)*
2. 🟠 **Управление объектами**: дать админу в UI кнопки «Редактировать / Снять с публикации / Удалить» для live‑объектов (API уже есть — открыть его админу с проверкой `role`).
3. 🟠 **Модерация отзывов**: новый `admin/reviews` (GET список + DELETE/hide) + вкладка.
4. 🟡 **Обзор просмотров / лидов / сделок** для админа (read + базовые действия).
5. 🟡 **Waitlist‑просмотр**, **broadcast‑уведомления**, **авторинг туров** — по мере надобности.
6. 🧹 Консолидировать две админки (`AdminPanel` vs `app/admin/*`) в одну.

*Ключевые файлы:* `components/homy/AdminPanel.tsx`, `app/admin/*`, `app/api/admin/users/route.ts`, `app/api/admin/listings/*`, `app/api/properties/[id]/route.ts`, `app/api/reviews/[id]/route.ts`, `lib/middleware/adminMiddleware.ts`, `lib/auth/userTypes.ts`.
