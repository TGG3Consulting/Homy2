# Homy — План к запуску: дорисовка экранов → бэк → миграция фронта

Основано на аудите (`COVERAGE-AUDIT.md`) и реальном коде (Next.js app-router, 28 роутов, 64 API). Скоуп: **весь продукт**. Исполнение: правки в реальном коде `C:\HomLy`. Брокер — **в запуск** (фичи + бэк).

Правило: без MVP, без костылей; каждый экран — рабочий, на реальных API, с состояниями (loading/empty/error). Ничего не выдумываем: где деталь неясна — фиксируем как ❓ и спрашиваем.

## 0. Честно про сроки
Полный объём (миграция 28 роутов + недостающие экраны + новый бэк под брокера + все состояния + QA) — **порядка 3–4 недель** одной нитью, не 1 день. Ниже план разбит на фазы и снабжён «критическим путём», чтобы можно было запускаться поэтапно. В конце — вопрос про дату/подскоуп.

## ФАЗА 2 — Бэк: докрытие логики под нарисованные экраны
(что нарисовано/нужно, но в API нет; первый шаг фазы — точная сверка эндпоинтов, затем стройка)

**Есть в API (проверено, экран просто подключаем):** auth (login/register/otp/reset), properties + [id]/(intelligence, reviews, virtual-tour, viewing-slots, nearby, opinion, chat), favorites, saved-searches, viewings (+approve/reject/propose/cancel), notifications, chats, support (inbox/assign/status), consultant/assigned, admin (dashboard/listings/users), users/me (+avatar, preferences, dashboard, recommendations, listings, properties), upload, search/voice.

**Нет в API — строим (схема БД + эндпоинты + роли/права):**
1. **Клиенты/CRM брокера** — `clients` (сущность, стадии лид/тёплый/холодный), CRUD `/api/clients`, привязка к брокеру.
2. **Сделки** — `deals` (пайплайн: переговоры/документы/закрыта/сорвана, сумма, комиссия), CRUD `/api/deals`.
3. **Заявки** — `inquiries` (входящие обращения по объектам), `/api/inquiries` (+ конверсия в клиента).
4. **Жизненный цикл объявления брокером** — редактирование/снятие: доработать `/api/properties/listings/[id]` (edit/unpublish), статусы.
5. **Ревизия ролей/прав** — гейтинг брокер/консультант/админ на новых эндпоинтах (middleware/guards).
6. **Миграции БД** + сиды тестовых данных под новые сущности.

> Точный список «есть/нет» подтверждаю чтением `app/api/**` перед стройкой — не по памяти.

## ФАЗА 3 — Миграция фронта (перенос макетов в React/Next)

**3.0 Фундамент (делается один раз, до экранов):**
- Дизайн-токены → в код (Tailwind config / CSS-переменные): цвета light/dark, Montserrat, радиусы, тени, изумруд.
- Ядро-компоненты по нашему дизайну: `Button` (стиль 20C), `Chip/FilterChip`, `Card` (объект), `Badge/StatusPill`, `Modal/Sheet`, `Nav` (лого-меню + «Назад», по правилу — на всех ролях), `Toast`, `EmptyState`, `Loader`, `ErrorState`, языковой переключатель.
- Утиль: тема (dark/light), i18n (RU/EN/HY) на существующем `useT`.

**3.1 Порядок миграции по роутам (dependency-driven):**
- **Ядро покупателя:** `/` → `/results` → `/properties/[id]` (шелл+под-вью) → `/allresults` → `/compare` → `/schedule`.
- **Авторизация:** `/login` → `/register`(+OTP) → `/forgot-password` → `/reset-password`.
- **Кабинет:** `/dashboard` (табы: обзор/избранное/просмотры/поиски/рекомендации/сообщения/**настройки**/поддержка).
- **Брокер (после бэка из Фазы 2):** `/dashboard/listings` → `/listings/new` → `/dashboard/properties` → `/dashboard/clients`(+new) → `/dashboard/inquiries` → `/dashboard/deals` → инбокс/статус консультанта.
- **Админ:** `/admin` → `/admin/moderation` → `/admin/users`.
- **Контент:** `/how-it-works` → `/about` → `/for-buyers` → `/for-owners` → `/for-renters` → `/list-property` → `/coming-soon`(по решению).

**3.2 На каждый экран (definition of done):**
данные с реального API · loading/empty/error · адаптив (моб/деск) · тёмная/светлая · i18n 3 языка · нав по правилу · доступность (фокус/контраст) · ручная проверка в браузере (скриншот) · нет console-ошибок.
